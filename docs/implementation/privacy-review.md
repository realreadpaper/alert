# 隐私审查记录

## 结论

当前设计与实现方向保持以下隐私边界：

- 不做账号。
- 不做云端服务。
- 不请求定位权限。
- 不上传设备状态。
- 不上传身份信息。
- 不保存历史轨迹。
- 看护判断在本机和组内设备之间完成。

## 数据分类

### 本机保存

| 数据 | 用途 | 存储位置 |
| --- | --- | --- |
| `deviceId` | 组内识别本机 | iOS Keychain / Android Keystore 保护的加密存储 |
| `groupSecret` | 生成 rollingId 和签名心跳 | iOS Keychain / Android Keystore 保护的加密存储 |
| 设备昵称 | 用户识别设备 | 本地加密配置 |
| 设备列表 | 首页展示与状态判断 | 本地加密配置 |
| 当前模式 | 外出、室内、静音阈值选择 | 本地配置 |
| 最近心跳状态 | 当前看护判断 | 本地运行状态 |

### 不保存

- 经纬度。
- 历史轨迹。
- 手机号。
- 通讯录。
- Apple ID。
- Google 账号。
- Wi-Fi SSID 明文。
- 长期原始 BLE 扫描记录。

### 不上传

当前实现没有云端 API、远程推送 API、分析 SDK 或广告 SDK。后续如加入任何网络能力，必须重新审查并更新本文档。

## 权限用途

### 蓝牙 / 附近设备

用途：发现看护组内手机是否仍在身边。

用户文案：

`用于发现看护组内手机是否仍在身边。`

### 通知

用途：设备可能离开时提醒用户。

用户文案：

`设备可能离开时，通知用于及时提醒你。`

### 本地网络

用途：同一 Wi-Fi 或热点下减少室内误报。

用户文案：

`同一 Wi-Fi 下可减少室内误报；不开启时仍可使用蓝牙看护。`

### Android 前台服务

用途：看护开启时维持本地检测，并显示系统要求的持续看护通知。

用户文案：

`开启持续看护时，系统需要显示看护状态通知。`

## 不请求定位权限

产品首版不做找回和地图能力，因此不应请求 iOS Location 或 Android Location 权限。BLE 权限实现应按系统版本选择 Nearby Devices / Bluetooth 权限，避免把定位权限作为默认依赖。

## 本地日志

允许保存：

- 最近连接状态摘要。
- 权限状态变化。
- 报警触发原因。
- 设备恢复连接时间。

禁止保存：

- 经纬度。
- 历史轨迹。
- 原始长期 BLE 扫描列表。
- 通讯录或账号信息。

日志导出必须由用户主动触发，并在导出前说明包含哪些本地诊断信息。

## 验证清单

在可安装 App 工程完成后，需要执行：

| 检查项 | 方法 | 通过标准 |
| --- | --- | --- |
| 无云端请求 | 抓包或系统网络监控 | 看护、配对、报警过程中无外部服务器请求 |
| 无定位权限 | 检查 Info.plist / AndroidManifest | 不声明定位权限 |
| 权限文案一致 | 对照系统权限弹窗和 App 内说明 | 权限用途与实际能力一致 |
| 本地密钥保护 | 检查 Keychain / Keystore 使用 | 密钥不落明文文件 |
| 数据清除 | 执行清除本地数据 | 组密钥和设备列表被删除 |
